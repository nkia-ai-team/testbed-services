# F21-Q / F21-P 머지 스펙 — Tomcat 200 포화 배관 확정본

정본 대비 실측 재검증 결과. 시트(`design-F21-threads-sheet.md`)와 초안 manifest/metadata의 코드
앵커는 grep으로 전부 재확인해 정확함을 확인했다. 아래는 **배선 단계에서 발견된 확정 편차**와
그대로 반영 가능한 JSON/코드 조각이다.

## 0. 확정 편차 요약 (중요도순)

1. **`prometheus.http_server_active_requests`는 존재하지 않는다 — 대체 지표로 교체 필수.**
   VictoriaMetrics(119:18428) `__name__` 전수 조회 결과 `http.server.active_requests`도
   Micrometer식 `tomcat_threads_busy_threads`류도 전혀 없다. Polestar APM 에이전트가 내는
   지표군은 `apm.agent.otel.java.{error_rate,error_count,percentile50/95/99,rps,span_count,
   jvm.thread.count,jvm.memory.*,jvm.cpu.*,db_responsetime}` 뿐 — in-flight 요청 수도
   Tomcat 스레드풀 전용 지표도 없다. 유일한 근사 후보는 `apm.agent.otel.java.jvm.thread.count`
   (`service_name`+`jvm_thread_state`+`jvm_thread_daemon` 라벨 보유, 실측 확인). 논데몬
   스레드 합계(`jvm_thread_daemon="false"`, 전 state 합산)를 Tomcat 워커+앱 스레드의 근사치로
   쓴다 — **실측 baseline: core-banking-api=52, food-delivery-order=65**(비교적 높은 baseline,
   포화 시 목표는 200 근접이므로 대략 3~4배 상승이 판정선). §2 NEW query로 반영.
2. **`prometheus.circuitbreaker_open`도 존재하지 않는다 — 제거, 기존 관측으로 대체.**
   `circuit`/`resilience` 계열 메트릭이 VM에 전무(resilience4j 메트릭 자체가 파이프라인에
   없음). 대신 CB open은 이미 승인된 `prometheus.apm_service_error_rate`(core-banking-api는
   이미 APPROVED_APM_SERVICES에 있음)의 급등으로 간접 관측 가능하다 — CB open이면 즉시 502가
   폭증하므로 `api_502_rate`의 기존 must_rule_out(`< 0.05`)이 이미 CB open 배제 역할을 겸한다.
   별도 query 불필요.
3. **`loadgen.food_create_status_rate` / `loadgen.transfer_2xx_rate`는 이미 존재하고 이미
   러너에 배선되어 있다**(`live_probes.py:495-509`, F20 작업으로 선반입됨). 단, **파라미터를
   받지 않는다** — 러너 코드가 `if query.parameters or query.query_id not in {...}: raise
   LiveProbeError("unsupported loadgen query")`로 **파라미터 존재 자체를 거부**한다. 초안
   manifest의 `"parameters": {"step": "create", "status_class": "503"}` 류는 전부 제거해야
   한다(팀장 브리핑의 "loadgen 관측 parameters 금지"와 정확히 일치). 필드 매핑은 고정:
   `loadgen.food_create_status_rate`→`business_5xx_rate`, `loadgen.transfer_2xx_rate`→
   `business_2xx_rate`.
4. **`service_name` 오탈자** — 초안 manifest가 food 관측에 `"food-order"`를 썼으나 실제
   APM 라벨은 `"food-delivery-order"`다(VM 실측 + `APPROVED_APM_SERVICES`/`APPROVED_HIKARI_SERVICES`
   양쪽 확인). `core-banking-api`는 이미 정확. 전체 food 관측의 service_name을
   `food-delivery-order`로 수정 필요.
5. **"무관 GET 지연 전염" 증거는 새 read-step 대신 기존 `prometheus.apm_service_p95` 재사용을
   권고**(편차). food entry의 `domain_profiles.read_step="menu"`는 **restaurant-service**를
   때리는 스텝(browsingJourney의 menu 조회)이지 order-service의 GET /api/orders가 아니다 —
   즉 같은 Tomcat 풀 전염을 증명하지 못한다. 신규 read-step 계측(GET /api/orders 태깅 +
   러너의 read_step latency 필드 신설)은 이 레포에 없는 north_south 실행기 스크립트
   (`profiles/load_north_south_executor.py`, 러너 클론에 부재 — tb-runner에만 배포된 것으로
   추정) 수정까지 필요해 배선 범위가 크다. 대신 **이미 승인된 `prometheus.apm_service_p95`**
   (food-delivery-order/core-banking-api 둘 다 APPROVED_APM_SERVICES에 기존재)를 서비스
   전체 p95로 재사용한다 — Tomcat 스레드 고갈은 정의상 서비스 전체(모든 엔드포인트)에
   전염되므로, 특정 read 엔드포인트를 분리하지 않아도 전체 p95 급등이 동일한 증거를 준다.
   단위 미확인(초/ms 불명, `percentile95` baseline 실측값 0 — 저트래픽 idle 상태에서 조회함)
   이라 정확한 임계치는 calibration에서 확정.
6. **F21-P 부하 형상**: 시트가 지적한 "250rps > max_rps 180" 문제는 **transfer 비중 상향
   변형 스크립트로 해결** — 새 `core-banking/loadgen/transfer-heavy-surge.js`(이체 40%,
   조회 35%, 내역 15%, 목록 10%)를 생성했다. `TARGET_RPS=90` 기본값이면 이체 rps
   ≈36(≥λ목표 25, 캘리브레이션 여유 확보), 총 90rps는 `target_rps.maximum=180`에 크게
   못 미쳐 상한 위반 없음. **매니페스트 초안의 `target_rps: 180`은 90으로 하향 정정.**
7. **F21-Q 부하 형상**: 신규 스크립트 불필요 — 기존 `order-surge.js`의 `TARGET_RPS`만
   상향하면 충분(§3 산술). **매니페스트 초안의 `target_rps: 55`는 70으로 상향 정정**
   (55×0.75=41 create/s로 λ목표 50 미달 — 70×0.75≈52.5로 목표 상회).
8. **워커 IP 매핑 재검증 완료, 스왑 없음.** 109 클러스터 `kubectl get nodes -o wide`
   실측: tb-w2=192.168.122.11(food-delivery), tb-w3=192.168.122.14(core-banking) —
   project-2025/CLAUDE.md 서버맵과 일치. host.stress `scenario_parameters`에 넣을 host는
   F21-Q="192.168.122.11", F21-P="192.168.122.14".
9. **러너 `APPROVED_K8S_TARGETS`에 banking api pod_ready 항목이 없다 — 신규 필요.**
   `("rca-testbed-food", "testbed-order")`는 F19-P/S로 이미 등록되어 있어 F21-Q는 추가 불요.
   그러나 `("rca-testbed-banking", "testbed-api")`는 부재. 109 실측(`kubectl get deploy
   testbed-api -n rca-testbed-banking -o jsonpath='{.spec.selector.matchLabels}'`) 결과
   `app=testbed-api` 확인 — §5 러너 조각 참조.
10. **host.stress 프로파일의 `allowed_scenarios`가 현재 `["F09-R","F05-P"]`뿐이고
    `location_strategy: "scenario"`라 F21용 `scenario_parameters`/`scenario_levels` 신설이
    필요**(시트가 이미 지적한 대로, main 반영 대상).
11. **db.workload는 `live_supported: false`(profiles.json 실측 확인)** — 시트의 판단 그대로,
    host.stress가 유일한 live 가능 지연 수단.
