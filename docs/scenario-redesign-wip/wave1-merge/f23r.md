# 머지 스펙 — F23-R 관측 배관 준비

정본: `docs/scenario-redesign-wip/design-F23-R-sheet.md` + `-manifest.json` + `-metadata.json`. 코드 앵커
(`ReconciliationBatch.java:46,74-91`, `InventoryService.java:79-83`, `ProductClient.java:25,37`,
`InventoryClient.java:33,48`) 재검증 완료 — 시트 그대로. 이 문서는 관측 query 3종 + env stopgap의
**실측 배선 초안**만 다룬다. 편집 금지 대상(registry/catalog/manifests/tests/profiles.py, 러너 레포)은
손대지 않았고, 이 문서는 그 편집을 맡을 사람을 위한 정확한 조각을 담는다.

## 0. 시트 대비 확정 편차 (중요 — 1건)

**시트 §4-A(3)의 전제가 부정확하다.** 시트는 "checkout_5xx_rate가 5xx 전용이라 409를 구조적으로 못
보고, script.js:228·surge.js:132가 409를 pass로 접어 emit 자체가 없다"고 적었다. 실측 결과:

- `script.js:228`·`surge.js:132`의 `check()` 판정(pass/fail bool)은 실제로 409를 "통과"로 접는다 — 이 부분은
  시트가 맞다.
- 그러나 **상태코드 자체는 이미 다른 경로로 emit되고 있다.** `scripts/scenarios/profiles/load_north_south_executor.py`
  의 `remote_script()`(라인 89-245)가 원격(tb-runner)에서 돌리는 python monitor는 k6의 `--out json=` 스트림을
  라인 단위로 파싱해 `http_reqs` 메트릭의 `tags.status`(실제 HTTP status, 정수)를 그대로 읽는다(라인 172-175).
  `check()` pass/fail이 아니라 **원시 상태코드**를 본다. 이 값으로 `business_2xx_rate`/`business_4xx_rate`
  (400~499 전체 뭉치)/`business_5xx_rate`를 계산해 `live.json`에 쓴다(라인 196-224). commerce의
  `business_step`은 `checkout`(profiles.json `load.north_south.parameter_contract.domain_profiles`
  `"http://192.168.122.77:30080"` 항목, 실측 확인).
- 즉 **409는 이미 `business_4xx_rate`(400+409 합산) 안에 잡혀 있다** — "emit 자체가 없다"는 시트 표현은
  과장이다. 진짜 갭은 더 좁다: **409를 400과 분리하는 전용 버킷이 없다.** 정상 케이스(쿠폰 오류 400)와
  장애 케이스(재고소진 409)가 `business_4xx_rate` 한 필드에 섞여 있어 감별에 못 쓴다.
- 수정 범위는 시트가 시사한 "loadgen 스크립트 보강"보다 **훨씬 작다**: `script.js`/`surge.js`의 check() 라벨은
  건드릴 필요 없고(이미 200/400/409를 pass로 접는 것과 무관하게 monitor가 원시 status를 본다), 고칠 지점은
  `load_north_south_executor.py`의 monitor 파이썬 블록(§2-C 조각 1곳) + `queries.json`/`live_probes.py`
  allowlist(§2-C 조각 2곳)뿐이다. 이 파일은 `scripts/scenarios/profiles/*.py`라 **내가 편집 금지** — 아래
  조각은 반영 담당자를 위한 정확한 diff다.
- 부수 확인: `load.north_south`의 `parameter_contract.allowed_scenarios`에 **F23-R·F19-Q 둘 다 아직 없음**
  (registry/profiles.json 실측, `'F23-R' in allowed_scenarios` → False). companion_refs 바인딩 전에 추가 필요.

## 1. 관측 query 3종 — 실측 확정

### 1-A. `database.inventory_stock_level` (신규)

명세대로 `database.index_present`(`INDEX_PRESENT_SQL`/`INDEX_PRESENT_CONTRACT`, live_probes.py:167-172,195-203)와
동일 어댑터(`database`) + `psql` 경유 패턴을 쓰되, 이 쿼리는 **런타임 스칼라 파라미터가 없다**(schema/table/agg가
전부 시나리오 고정 상수) — 오히려 `PG_SLOW_ACTIVE_QUERY_SQL`(live_probes.py:179-194, 파라미터 없이 고정 SQL +
contract는 allowlist 매칭 전용)과 동형이다.

**SQL (신규 상수 `INVENTORY_STOCK_LEVEL_SQL`)**:
```sql
SELECT count(*) AS zero_stock_count FROM inventory_schema.inventory WHERE stock = 0
```

**실측 (2026-07-24, tb-cp `testbed-postgres-0`, `psql -U commerce -d commerce`)**:
```
SELECT min(stock), avg(stock), count(*) FILTER (WHERE stock=0), count(*) FROM inventory_schema.inventory;
→ min=0 | avg=79.0 | zero_stock_count=1 | total=16
```
16개 flagship 상품 중 1개가 현재 stock=0(정상 운영 중 자연 소진분 — 배치가 살아있어 톱니파의 저점). 쿼리가
스키마·값과 정합됨을 확인. `agg` 파라미터는 매니페스트가 이미 `count_zero_stock`로 지정했으므로 이 SQL로 확정.

**Contract 상수 (신규)**:
```python
INVENTORY_STOCK_LEVEL_CONTRACT = {
    "db_host": "192.168.122.77",
    "db_port": 30432,
    "db_name": "commerce",
    "db_user": "commerce",
    "schema": "inventory_schema",
    "table": "inventory",
    "agg": "count_zero_stock",
}
```

**`_database_observation` 분기 (live_probes.py, `database.index_present` 분기 바로 위/아래에 삽입)**:
```python
if query.query_id == "database.inventory_stock_level":
    if dict(query.parameters) != INVENTORY_STOCK_LEVEL_CONTRACT:
        raise LiveProbeError("inventory stock level target is not allowlisted")
    response = self.database_client(
        INVENTORY_STOCK_LEVEL_SQL, (), credentials=self.database_credentials,
    )
    count = response.get("zero_stock_count")
    if isinstance(count, bool) or not isinstance(count, int) or count < 0:
        raise LiveProbeError("inventory zero-stock count is invalid")
    return count, _response_time(response, self.clock), "database:inventory-stock-level"
```

**`production_runtime.py` `database_client` 분기 (psql 경로, 파라미터 없음 — `PGOPTIONS` 불필요)**:
```python
elif sql == INVENTORY_STOCK_LEVEL_SQL and parameters == ():
    pgoptions = ""
    result_field = "zero_stock_count"
```
(`result_field`가 `"zero_stock_count"`인 분기를 `value = int(raw)`로 떨어지는 기존 else 경로에 태우면 됨 —
`index_present`처럼 별도 bool 변환 불필요.)

주의: 매니페스트의 `success.stock-depleted`(`inventory_stock_level >= 1`)와 감별표의 "단조감소→0"은 이
count가 **낮을수록 정상, 높을수록(특히 16 전체 소진) 장애 근접**이라는 뜻이므로 값 해석 방향을 헷갈리지
않게 sheet §2 감별표에 "count_zero_stock 임계는 절대값보다 **추세**(단조증가 vs 톱니파 진동)로 판정"이라는
주석을 다는 편이 안전하다(현재 manifest에는 이 뉘앙스가 없음 — 후속 보강 후보).

### 1-B. `database.restock_movement_rate` (신규)

**SQL**:
```sql
SELECT count(*) AS restock_count FROM inventory_schema.inventory_movements
WHERE movement_type = 'RESTOCK' AND created_at >= now() - interval '12 minutes'
```
`window` 파라미터(매니페스트 `"12m"`)는 SQL 리터럴에 굳이 바인딩할 필요 없이 고정(F19-Q의
`dispatch_delivered_rate` 동형 쿼리도 미구현 상태라 대응 패턴 없음 — 이 조각이 그 계열의 첫 구현이 된다).

**실측 (2026-07-24, 동일 접속)**:
```
movement_type 분포: ADJUST=26, RELEASE=70847, RESERVE=743734, RESTOCK=5022
최근 12분 RESTOCK count = 4
```
배치가 정상 동작 중(10분 주기)이라 4건이 관측된 것으로, 정지 시 이 값이 0으로 떨어지는 것이 결정 증거라는
설계가 실측과 일치한다.

**Contract 상수**:
```python
RESTOCK_MOVEMENT_RATE_CONTRACT = {
    "db_host": "192.168.122.77",
    "db_port": 30432,
    "db_name": "commerce",
    "db_user": "commerce",
    "schema": "inventory_schema",
    "table": "inventory_movements",
    "movement_type": "RESTOCK",
    "window": "12m",
}
```

**`_database_observation` 분기**:
```python
if query.query_id == "database.restock_movement_rate":
    if dict(query.parameters) != RESTOCK_MOVEMENT_RATE_CONTRACT:
        raise LiveProbeError("restock movement target is not allowlisted")
    response = self.database_client(
        RESTOCK_MOVEMENT_RATE_SQL, (), credentials=self.database_credentials,
    )
    count = response.get("restock_count")
    if isinstance(count, bool) or not isinstance(count, int) or count < 0:
        raise LiveProbeError("restock movement count is invalid")
    return count, _response_time(response, self.clock), "database:restock-movement-rate"
```
`production_runtime.py`에 동일 형태의 `elif sql == RESTOCK_MOVEMENT_RATE_SQL and parameters == ():` 분기 추가.

### 1-C. `loadgen.checkout_status_rate` (409 버킷) — §0 편차 반영, 축소된 실제 작업

시트가 제안한 "script.js/surge.js가 status-class를 emit하도록 보강"은 **불필요**(§0). 실제 필요 작업은 3곳,
전부 `load_north_south_executor.py`의 monitor 블록(라인 196-224 부근) + allowlist 2곳:

**monitor 블록 확장 (라인 205 이후, `business_5xx_rate` 계산 옆에 추가)**:
```python
business_409_rate = (
    sum(status == 409 for _, status in checkout_results) / checkout_count
    if checkout_count else 0.0
)
```
그리고 `document = {...}` 딕셔너리(라인 213-225)에 `"business_409_rate": business_409_rate,` 한 줄 추가.
이렇게 하면 400(쿠폰 등)과 409(재고소진)가 분리되고, `business_4xx_rate`(합산)는 하위호환으로 유지된다.

**`observation_queries.json`에 신규 query_id**:
```json
"loadgen.checkout_status_rate": {
  "adapter": "loadgen_summary",
  "selector": "checkout.409.rate",
  "freshness_sec": 30,
  "allowed_parameters": ["step", "status_class"]
}
```
(매니페스트가 이미 `{"step": "checkout", "status_class": "409"}`로 파라미터를 못박아 뒀으므로 그대로.)

**`live_probes.py` `_loadgen_observation` allowlist + field map 확장**:
```python
"loadgen.achieved_rps", "loadgen.checkout_5xx_rate",
"loadgen.write_step_status_rate", "loadgen.read_step_status_rate",
"loadgen.food_create_status_rate", "loadgen.transfer_2xx_rate",
"loadgen.checkout_status_rate",   # 신규
```
```python
"loadgen.checkout_status_rate": "business_409_rate",   # 신규 field map
```
`field.endswith("_rate")`이므로 기존 range-check([0,1])가 그대로 적용됨 — 추가 분기 불필요.

이 세 곳 모두 **금지 경로**(`scripts/scenarios/profiles/*.py`, 러너 레포)에 있어 내가 직접 편집하지
않았다. 위 조각은 반영 담당자에게 그대로 넘길 수 있는 최종 diff다.

## 4. 남는 갭 (이 배관 이후에도 blocked 유지 사유)

1. **§1-C의 세 조각**(`load_north_south_executor.py` 편집 + `observation_queries.json` 등록 + `live_probes.py`
   allowlist)이 미반영 상태에서는 `loadgen.checkout_status_rate` query가 여전히 `LiveProbeError`로 거부된다.
2. **§4-B 런타임 스케줄러 정지 훅**(business.fault를 실제 injector로 승격)은 이 배관과 별개 트랙 — 이 문서는
   손대지 않았다(코드 executor 신설은 scope 밖).
3. `business.fault` 프로파일의 `parameter_contract`가 현재 **아예 없음**(§ 위 실측) — F23-R/F19-Q
   `scenario_parameters` 키잉·`allowed_scenarios`는 훅 신설과 함께 추가돼야 하며, 이 문서의 §1/§2/§3은
   그 전 단계(관측·stopgap 실측)만 담당한다.
