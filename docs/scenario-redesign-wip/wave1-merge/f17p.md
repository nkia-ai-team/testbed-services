# 머지 스펙 — F17-P dual-arm 배관 준비

정본: `docs/scenario-redesign-wip/design-F17-P-sheet.md` + `-manifest.json` + `-metadata.json`. 코드 앵커
(`TransferService.java:52-89`, `AccountService.java:57-81`, `k8s/22-transfer-service.yaml:105-120`,
`db/init.sql:91-92`) 재검증 완료 — 시트 그대로 정확하다. 이 문서는 dual-arm 실행 가능성 판정 +
관측 배선 실측 + controller 초안 + cleanup 등급 판정을 다룬다. 편집 금지 대상(registry/catalog/
manifests/tests/profiles/*.py, 러너 레포)은 손대지 않았고, 아래는 그 편집을 맡을 사람을 위한
정확한 조각이다. **본인 전용 신규 파일**로 `core-banking/loadgen/frozen-bypass.js`를 작성했다
(team-lead 프롬프트는 `commerce/loadgen/frozen-bypass.js`를 지시했으나, 시트·매니페스트가
`script_path: /opt/loadgen/core-banking/frozen-bypass.js`로 못박아 뒀고 기존 디렉터리 관례
(`core-banking/loadgen/surge.js` 등)와도 일치해 정본을 따랐다 — §0-1 참조).

## 0. 시트/매니페스트 대비 확정 편차

### 0-1. 스크립트 위치: `core-banking/loadgen/`(team-lead 지시 `commerce/loadgen/`과 다름)
매니페스트의 `script_path`가 `/opt/loadgen/core-banking/frozen-bypass.js`로 고정돼 있고, 배포 스크립트가
도메인 디렉터리→`/opt/loadgen/<domain>/` 미러링 관례를 따르므로(`core-banking/loadgen/surge.js` 등 기존
파일 전부 이 패턴) `core-banking/loadgen/frozen-bypass.js`가 맞다. team-lead 프롬프트의 `commerce/loadgen/`
표기는 오타로 판단 — 시트가 authoritative이므로 그대로 따랐다.

### 0-2. `load.north_south` 파라미터 계약은 **정확히 9개 키만 허용** — 매니페스트 초안은 계약 위반
`load_north_south_executor.py:26-31`(`validate_parameters`)가 `set(parameters) != required`이면
즉시 `ExecutorError`를 던진다. `required = {target_rps, ramp_up, hold, ramp_down, entry_url, script_path,
scenario_tag, seed, baseline_unit}` — **정확히 9개**. 매니페스트 초안(`design-F17-P-manifest.json:108-124`)의
`control_entry_url`/`frozen_from_account`/`closed_to_account`/`active_counterparty`/`amount_min`/`amount_max`
6개 키는 이 계약을 통과하지 못한다. **해결**: food-delivery의 `order-surge.js` 선례(`RESTAURANT_URL`/
`DISPATCH_URL`을 실행기가 안 넘기는 스크립트 내부 상수로 하드코딩)를 그대로 따라 6개 값 전부를
`frozen-bypass.js` 내부 상수로 옮겼다. `entry_url` 하나(direct arm의 30282)만 계약 파라미터로 남고,
control arm URL·계좌 ID·금액 범위는 전부 스크립트 상수다. **controller 블록(§3)의 `profile.levels[0].parameters`는
9키로 축소해야 한다.**

### 0-3. 관측 3종 중 2종은 **신규 query_id가 필요 없다** — 시트 §4-2의 전제가 과장
시트는 `loadgen.frozen_bypass_completed_rate`/`loadgen.normal_path_reject_rate` 둘 다 "NEW query_id" +
"이 계약 안에 배선 없음"이라 적었다. 실측(`live_probes.py:494-515` `_loadgen_observation`)하면:
- `document`(=k6 monitor의 `live.json`)는 이미 `business_2xx_rate`/`business_4xx_rate`/`business_5xx_rate`/
  `business_nonok_rate`/`read_2xx_rate`/`read_nonok_rate`를 **범용 필드명**으로 싣는다
  (`load_north_south_executor.py` 원격 monitor.py, 라인 196-224) — `business_step`/`read_step`이 어떤
  문자열이든 상관없이 태그만 일치하면 채워진다.
- 이미 등록된 범용 query_id `loadgen.transfer_2xx_rate`(selector `business.2xx.rate` → field
  `business_2xx_rate`)와 `loadgen.read_step_status_rate`(selector `read.nonok.rate` → field
  `read_nonok_rate`)를 **그대로 재사용**하면 된다 — `live_probes.py`의 allowlist(494-499)·field map
  (502-509)에 이미 있어 **러너 코드 변경 0줄**.
- 필요한 건 오직 `registry/profiles.json`의 `domain_profiles`에 30282용 신규 항목을 추가해
  `business_step="frozen-direct"`(direct arm 태그), `read_step="frozen-control"`(control arm 태그, "read"는
  아니지만 두 번째 집계 슬롯을 재활용)로 지정하는 것뿐이다. **진짜 신규가 필요한 건 §1-C
  (`database.integrity_violation_count`) 하나뿐이다** — 시트가 3종 전부 신규라 한 것보다 실제 갭이 좁다.
- 정확도 한계(반드시 문서화할 것): `business_2xx_rate`는 HTTP status만 보고 응답 바디를 보지 않는다.
  `TransferService.execute()`는 COMPLETED든 FAILED든 **항상 HTTP 200**을 반환한다(`TransferResponse`
  본문의 `status` 필드만 다르다) — 따라서 `frozen_bypass_completed_rate`(=`business_2xx_rate` 재사용)는
  "direct arm 요청이 (계좌 상태와 무관하게) 200을 받았다"는 것만 증명하고 "실제 COMPLETED로 확정됐다"는
  것까지는 증명하지 못한다. **결정적 증거는 여전히 `integrity_violation_count`(DB ground-truth)다** — 성공
  조건에서 이 두 필드는 보조 신호로, DB count가 1차 증거로 유지돼야 한다(§4에 반영).

### 0-4. 잔액 고갈·공유 baseline 오염 — 시트·매니페스트에 없던 신규 발견
매니페스트 초안(`target_rps:20, hold:8m`)대로 direct arm을 매 iteration 흘리면:
- variant A(FROZEN `ACC-9001`에서 출금): 잔액 100000, 평균 이체액 3000 가정 시 **초 단위**(10rps 기준
  ~3.3초)로 고갈. 고갈 후에도 응답은 여전히 200(FAILED 본문)이라 §0-3의 정확도 한계와 겹쳐 관측이
  더 흐려진다.
- variant B(CLOSED `ACC-9002`로 입금, 자금원=`ACTIVE_COUNTERPARTY` `ACC-1001`): `ACC-1001`은
  **surge.js의 `ACTIVE_ACCOUNTS` 풀에 속한 공유 baseline 계좌**(`core-banking/loadgen/surge.js:69`)다.
  8분 내내 흘리면 5,000,000 잔액이 수 분 내 고갈되고, 이 시나리오가 끝난 뒤에도 baseline·다른
  시나리오의 이체가 잔액부족으로 **조용히 실패**하는 오염을 남긴다 — `project-2025/CLAUDE.md`에 기록된
  `commerce-settlement` 계좌 고갈 사례(5일간 27.5만 건 조용한 FAILED)와 동형 위험.
- **조치(이미 반영, `core-banking/loadgen/frozen-bypass.js` 참조)**: `DIRECT_ARM_RATIO=0.02`로 direct arm
  발생을 억제 — control arm(잔액 영향 없음, account-service가 마주치기 전에 400 거절)은 매 iteration
  그대로 흘려 `normal_path_frozen_reject_rate`(read_nonok_rate) 물량을 확보한다.
  `integrity_violation_count`는 절대값 count>0(단조증가, 1건만 있어도 성공조건 영구 충족)이라 낮은
  비율로도 충분하다. 잔존 위험(잔액 일부 소모, 완전 무해화 아님)은 §5 cleanup 판정에 반영했다.

## 1. dual-arm 실행 가능성 — 판정: **가능(계약 확장 없이, 스크립트 내부 하드코딩으로)**

`load.north_south`가 "단일 entry_url·단일 스크립트"만 전제한다는 우려(원 프롬프트)는 절반만 맞다 —
계약 파라미터 레벨에서는 URL 하나만 받지만, **스크립트가 두 번째 URL을 내부 상수로 하드코딩**하는 건
food-delivery `order-surge.js`(`RESTAURANT_URL`/`DISPATCH_URL` 3-URL 패턴, executor는 `ORDER_URL` 하나만
주입)에 이미 선례가 있다. F17-P도 동일 패턴: `entry_url`=direct arm(30282), control arm(30082)은 스크립트
상수. **전용 one-shot HTTP injector 프로파일 신설은 불필요** — 시트 §4의 "설계 판단 필요" 항목에 대한 답.

`business_step`(monitor 집계용) 태그: `frozen-direct`(direct arm), `frozen-control`(control arm, read_step
슬롯 재활용). §0-3에서 확인했듯 기존 range-check 로직(`field.endswith("_rate")`→[0,1] 검증)이 그대로
적용되므로 추가 분기 불필요.

### 2-A. `database.integrity_violation_count` (신규) — `outbox_unpublished_count`와 동형

`live_probes.py:905-918`의 `database.outbox_unpublished_count` 분기(고정 SQL + `OUTBOX_UNPUBLISHED_CONTRACT`
allowlist, `kubectl exec testbed-oracle-0` + `sqlplus`)와 동일 패턴. 시트 원안의 `:since` 바인드 파라미터는
불필요 — surge.js baseline이 `ACC-9001`/`9002`를 의도적으로 제외하므로(§1 근거, surge.js:69 주석) 평시
이 조건을 만족하는 행이 원천적으로 없다. 따라서 시간창 없는 **절대 count**로 충분하고 단순하다.

**Contract 상수 (신규, `outbox_unpublished_count` 옆에 추가)**:
```python
INTEGRITY_VIOLATION_CONTRACT = {"namespace": "rca-testbed-banking"}
```

**SQL (신규 상수)**:
```sql
SELECT count(*) FROM banking.transfers t
JOIN banking.accounts a ON a.id IN (t.from_account, t.to_account)
WHERE a.status IN ('FROZEN','CLOSED') AND t.status = 'COMPLETED'
```

**`_database_observation` 분기 (live_probes.py, `outbox_unpublished_count` 분기 바로 아래 삽입)**:
```python
if query.query_id == "database.integrity_violation_count":
    # F17-P 결정적 증거: 5xx가 전혀 없는 무결성 위반이라 이 DB count가 유일한
    # ground-truth witness다(design-F17-P-sheet.md §2). surge.js baseline이
    # ACC-9001/9002를 의도적으로 제외하므로(surge.js:69) 평시 이 count는 항상 0.
    if dict(query.parameters) != INTEGRITY_VIOLATION_CONTRACT:
        raise LiveProbeError("integrity violation target is not allowlisted")
    result = self._kubectl(
        "exec", "testbed-oracle-0", "--namespace", "rca-testbed-banking", "--",
        "sh", "-lc",
        "printf 'alter session set container=FREEPDB1;\\nset pages 0 feedback off heading off\\n"
        "select count(*) from banking.transfers t join banking.accounts a "
        "on a.id in (t.from_account, t.to_account) "
        "where a.status in (chr(39)||chr(70)||chr(82)||chr(79)||chr(90)||chr(69)||chr(78)||chr(39),"
        "chr(39)||chr(67)||chr(76)||chr(79)||chr(83)||chr(69)||chr(68)||chr(39)) "
        "and t.status=chr(67)||chr(79)||chr(77)||chr(80)||chr(76)||chr(69)||chr(84)||chr(69)||chr(68);\\n"
        "exit;\\n' | sqlplus -s / as sysdba",
    )
    raw = result.stdout.strip()
    if not re.fullmatch(r"[0-9]+", raw):
        raise LiveProbeError("integrity violation count is invalid")
    return int(raw), _aware(self.clock()), "database:integrity-violation-count"
```
(`chr()`로 문자열 리터럴을 우회하는 건 `ORACLE_TAG_CONTRACT` 분기의 기존 관례를 따른 것 — 작은따옴표
포함 SQL을 쉘 이스케이프 없이 안전하게 넘기기 위함. 실제 `sqlplus` 접속 계정에 `banking` 스키마 조회
권한이 있는지는 배포 시점 확인 필요 — `outbox_unpublished_count`가 이미 같은 스키마를 쓰므로 있을 것으로
추정.)

**`registry/queries.json` 신규 항목**:

## 5. cleanup — 판정: **3등급 이관 필요, F17-P 보류 권고**

시트 §4-3이 요구한 "우회 arm이 남긴 위반 row 원상복구"를 실측 대조한 결과, **현재 러너 인프라에는 쓰기
가능한 executor 등급이 아예 없다**:
- `rca-scenario-runner/backend/app/live_probes.py`의 모든 `_database_observation` 분기(F01-P 세션수,
  MySQL index, outbox count, F17-P 신규 포함)는 예외 없이 `SELECT`뿐이다. `production_runtime.py`의
  `database_client`도 동일(§2-A 확인) — **UPDATE/DELETE를 실행하는 코드 경로가 없다.**
  `registry/controllers.json` 전체를 스캔해도 `dispatcher_mode`는 `"trusted"` 하나뿐 — 관측/부하 실행기
  이외의 "쓰기 정정" 등급이 애초에 정의돼 있지 않다.
- F17-P의 direct arm이 남기는 흔적은 두 겹이다: (a) `transfers` 행 자체(COMPLETED, FROZEN/CLOSED 당사자) —
  이건 오히려 **캡처의 결정적 증거이므로 캡처 export 완료 전까지는 지워서도 안 된다**(§4의 `capture`
  블록과 순서 상충 위험, cleanup은 반드시 `export_not_before` 이후). (b) 계좌 balance 변동(`ACC-9001`
  차감, `ACC-9002`/`ACC-1001` 증감) — 원장 이벤트가 outbox→relay→Kafka로 비동기 발행되므로(`TransferService.
  java:100-104`, `db/init.sql` outbox 주석), DB 레벨에서 balance를 되돌려도 이미 Kafka로 나간 이벤트를
  소비한 하류(ai-observer 등)까지 소급 정정되지 않는다 — **완전한 원상복구는 DB 정정만으로도 불가능**하다.
- 결론: cleanup을 자동화하려면 (i) 신규 쓰기(UPDATE) executor 등급을 러너에 신설하고, (ii) capture
  export 완료 이후에만 발동하도록 순서를 재설계하고, (iii) Kafka 하류 소급 정정은 별도로 범위 밖 선언해야
  한다 — 이 세 가지 모두 이 세션의 스코프(관측 배관 준비) 밖이다. **team-lead가 제시한 "3등급 이관"
  판정을 그대로 적용한다: cleanup 수단이 없으므로 F17-P는 live 승격을 보류한다.**

**§0-4에서 반영한 `DIRECT_ARM_RATIO=0.02`는 잔존 영향을 줄일 뿐 cleanup 문제를 해소하지 않는다** — 낮은
비율이라도 `capture`가 `create_golden_anomaly: false`(evaluation/calibration run)인 draft 단계에서 반복
실행하면 `ACC-9001`/`ACC-1001`/`ACC-9002` 잔액이 누적 변동된다. 쓰기 executor가 없는 한 이 누적은 수동
SQL 정정(`kubectl exec` + `sqlplus`, 사람이 직접)에 의존해야 한다.

## 6. 종합 판정

| 골든 4조건 | 판정 | 근거 |
|---|---|---|
| ① 코드 앵커 | ✅ | 시트 그대로, 재검증 완료 |
| ② 정답(answer-key) | ✅ | 메타데이터 그대로 유효 |
| ③ 감별 가능 | 🟢 (기존 대비 개선) | 3종 중 2종은 신규 코드 0줄(기존 query_id 재사용), 1종만 신규(§2-A, `outbox_unpublished_count`와 동형이라 저위험). 단 `frozen_bypass_completed_rate`의 body-레벨 한계(§0-3)는 success 기준에서 제외해 보완(§4) |
| ④ 주입 수단 | ✅ (계약 확장 없이 스크립트 하드코딩으로 해결, §1) | `core-banking/loadgen/frozen-bypass.js` 작성 완료, `DIRECT_ARM_RATIO`로 잔액·공유계좌 리스크 완화(§0-4) |
| **cleanup** | ❌ **블로커 — 3등급(쓰기 executor) 자체가 러너에 없음** | §5 |

**승격 판정: 보류(draft 유지, live_allowed=false 그대로).** ①②④는 이번 세션에서 완전히 해소했고 ③도
시트의 우려보다 훨씬 좁혀졌지만, cleanup은 이 리포지토리·러너 어느 쪽에도 쓰기 executor 등급이 없어
자동화가 원천적으로 불가능하다 — 이는 F17-P 한 건의 문제가 아니라 "DB mutation을 남기는 모든 P7류
시나리오"에 공통될 인프라 갭이므로, 팀 차원에서 쓰기 executor 등급 신설을 별도 트랙으로 먼저 결정한
뒤 재승격을 시도할 것을 권고한다.
