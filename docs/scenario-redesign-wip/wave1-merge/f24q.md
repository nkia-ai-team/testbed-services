# F24-Q 머지 스펙 — restaurant 상류차단 → order 전량 502 배관 확정본

정본(`design-F24-Q-sheet.md`/`-manifest.json`/`-metadata.json`) 대비 실측 재검증 결과. 코드
앵커는 전부 grep으로 재확인해 정확함을 확인했다. 그러나 **초안 manifest의 관측 배선 3곳이
러너 현재 코드로는 그대로 반영 불가능한 상태**였다 — 아래 §0이 확정 편차, §1이 재검증된
앵커, §2~§6이 반영 가능한 조각이다.

## 0. 확정 편차 요약 (중요도순)

1. **주입 수단은 F02-P 경로 그대로 재사용 가능 — 확인.** `scripts/scenarios/registry/profiles.json`의
   `load.north_south.scenario_parameters.F02-P`가 이미 `entry_url: http://192.168.122.77:30181`
   (restaurant NodePort), `script_path: /opt/loadgen/food-delivery/surge.js`,
   `baseline_unit: loadgen-food`, `target_rps: 50`으로 F02-P의 **companion** 프로파일로
   restaurant를 흘리고 있다. F24-Q는 이 프로파일을 **primary**(단독, db.ddl 없이)로 승격
   반영만 하면 된다 — 신규 executor·스크립트 전혀 불필요. `target_rps=70`(초안)은 F02-P의
   50보다 높은 별도 값으로 그대로 채택 가능(달리 산정할 근거 없으므로 calibration 대상으로
   유지).
2. **`prometheus.circuitbreaker_open`은 존재하지 않는다 — 폐기, 팀장 브리핑과 일치.**
   `observation_queries.json`의 `queries` 키 전체를 열거해 확인 — `circuitbreaker` 계열
   query_id가 레지스트리에 전무하다. 초안 manifest의 `restaurant_cb_open` 관측(및
   metadata의 "(관측 배선 시) CB open" 문구)은 완전히 제거한다. 대체 결정 증거는 이미
   설계된 대로 `restaurant_error_rate`/`restaurant_p95` 급등 + `payment_502_rate` 낮음의
   조합으로 충분하다(F19-S·F21도 동일하게 CB 신호를 error_rate로 대체).
3. **`http.target_health`는 파라미터를 전혀 받지 않고, 고정 URL(commerce gateway 30080)만
   본다 — restaurant 전용 target_health는 원천 불가능.** `observation_queries.json`에
   `"http.target_health": {..., "allowed_parameters": []}`로 명시돼 있고, 러너
   `live_probes.py`의 `_http_observation`이 `if query.parameters: raise LiveProbeError
   ("unsupported http query")`로 파라미터 존재 자체를 거부한다. 실제 호출도
   `TARGET_HEALTH_URL = "http://192.168.122.77:30080/health"`(commerce gateway) 하나로
   하드코딩돼 있어 도메인 불문 전역 신호다. 초안 manifest의 `restaurant_target_health`
   (`parameters: {service, namespace, port, path}`)는 **그대로 반영하면 러너가 즉시
   LiveProbeError로 거부한다** — 삭제하고, restaurant 자체 헬스는 `restaurant_pod_ready`
   (kubernetes 어댑터, 아래 참조)와 `restaurant_error_rate`의 회복으로 대체한다.
   `entry_status`(`http.entry_health`, 전역 엔트리 생존)는 기존 그대로 유지.
4. **`loadgen.food_create_status_rate`는 파라미터를 받지 않고 단일 집계 필드
   (`business_5xx_rate`)만 반환한다 — 502/503 분리는 이 query로 원천 불가능.**
   `_loadgen_observation`이 `if query.parameters or query.query_id not in {...}: raise
   LiveProbeError("unsupported loadgen query")`로 파라미터 자체를 거부하고, 매핑도
   `"loadgen.food_create_status_rate": "business_5xx_rate"` 단일 필드뿐이다. 초안의
   `order_create_502_rate`(`status_class=502`)·`order_create_503_rate`(`status_class=503`)
   두 관측은 **둘 다 파라미터 포함이라 그대로 반영 불가** — 팀장 브리핑("loadgen 관측은
   parameters를 받지 못함")과 정확히 일치한다. 단일 `order_create_5xx_rate`(파라미터 없음)로
   교체한다. F24-Q(502) vs F19-Q(503) 감별은 **애초 설계 의도대로** loadgen 신호가 아니라
   restaurant/dispatch 각각의 APM error_rate·p95로 이뤄져야 한다 — sheet §2의 감별표 자체는
   이미 downstream별 신호를 결정 증거로 삼고 있어 설계 의도와 충돌 없음, 다만 metadata의
   "order_create_502_rate ≥ 0.1"이라는 must_support 문구는 "order_create_5xx_rate"로 정정
   필요.
5. **`APPROVED_APM_SERVICES`에 `food-delivery-restaurant`가 없다 — sheet/manifest의
   "이미 등록·라이브(F02-P가 실사용)" 주장은 틀렸다.** `scripts/scenarios/registry/
   controllers.json`의 F02-P 컨트롤러는 실제로 `restaurant_p95`
   (`service_name=food-delivery-restaurant`) 관측을 갖고 있고 `live_enabled: true`이지만,
   러너 `backend/app/live_probes.py`의 `APPROVED_APM_SERVICES` frozenset에는
   `food-delivery-restaurant`가 전혀 없다(`commerce-*`, `food-delivery-payment`,
   `food-delivery-order`, `core-banking-*`만 있음). 러너 저장소 전체 git 히스토리에서도
   `food-delivery-restaurant` 문자열이 한 번도 등장한 적이 없다(`git log --all -S`로 확인).
   즉 **F02-P의 restaurant_p95 관측은 현재 라이브에서 매 tick LiveProbeError를 내고 있을
   가능성이 높다** — F24-Q 착수 전에 반드시 고쳐야 할 선행 결함이며, F24-Q도 동일 관측을
   재사용하므로 반드시 같이 추가해야 한다(§3).
6. **F02-P의 `pod_ready` 관측도 `resource` 파라미터가 빠져 있어 같은 계열의 잠재
   버그로 보인다(참고 보고, 본건 직접 수정 대상 아님).** `controllers.json`의 F02-P
   `pod_ready`는 `parameters: {"namespace": "rca-testbed-food"}`만 주고 `resource`가
   없다 — 러너 `_kubernetes_observation`은 `resource` 미지정 시 기본값
   `"api-gateway"`로 채워 `("rca-testbed-food", "api-gateway")` 키로 조회하는데 이는
   `APPROVED_K8S_TARGETS`에 존재하지 않아 역시 LiveProbeError가 예상된다. F24-Q는 이
   실수를 반복하지 않도록 `resource: "testbed-restaurant"`를 명시한다(§3).
7. **restaurant pod_ready 타깃은 `APPROVED_K8S_TARGETS`에 없다 — 신규 1행 필요.**
   `("rca-testbed-food", "testbed-restaurant")` 항목이 없다. 109 k8s manifest 실측
   (`food-delivery/k8s/21-restaurant-deploy.yaml:7,12,16,100,114` 전부 `app:
   testbed-restaurant`)으로 라벨 `app=testbed-restaurant` 확인 — §3 반영.
8. **앵커 재검증 결과 전부 일치.** `OrderService.java:57`(`@Transactional createOrder`),
   `:60`(`getRestaurant`), `:73`(`getMenu`) 확인. `RestaurantClient.java:29-30`
   (`@CircuitBreaker`+`@Retry` on getRestaurant), `:58-59`(동일 on getMenu), `:44`
   (getRestaurantFallback 502), 코드 재확인 결과 getMenuFallback 502는 라인
   `:80` 부근(`getMenuFallback` 시그니처가 `:79`, 본문 502는 `:81` 근방 — 시트의 `:55`
   표기는 실제로는 getRestaurantFallback(`:50-56`) 범위 안에 있음, 확인 결과 큰 편차
   아니나 metadata `code_anchor`의 정확한 라인은 `:44`(getRestaurantFallback 502),
   `getMenuFallback`은 `RestaurantClient.java:79-84` 범위로 소폭 정정 권고). order
   `application.yml:54`(restaurant read-timeout 5s) 확인, `:69`(CB restaurant
   sliding-window-size 10), `:96,98-100`(retry restaurant max-attempts 3·wait-duration
   200ms) 확인 — 라인 번호가 시트의 `:98-104`와 약간 다르나(`:96` retry 섹션 시작,
   `:98` max-attempts) 실질 내용은 100% 일치. restaurant `application.yml:13`
   (Hikari maximum-pool-size 10) 확인. `k8s/21-restaurant-deploy.yaml:118`
   (nodePort 30181) 확인.

## 2. observation_queries.json — 신규 query 불필요

F24-Q에 필요한 관측은 전부 기존 query_id로 충분하다. **신규 query_id 추가 없음.**
사용할 기존 query: `http.entry_health`, `kubernetes.pod_ready`,
`prometheus.apm_service_p95`, `prometheus.apm_service_error_rate`,
`loadgen.food_create_status_rate`(파라미터 없이).

## 7. metadata 정정 (design-F24-Q-metadata.json 대비)

- `must_support`의 `"order create 502 비율 급증 (order_create_502_rate ≥ 0.1)"` →
  `"order create 5xx 비율 급증 (order_create_5xx_rate ≥ 0.1) — 502/503 분리는 이 신호로
  불가, restaurant/dispatch 각각의 APM 신호로 감별"`로 정정.
- `must_support`의 `"(관측 배선 시) order측 resilience4j cb=restaurant open"` 항목 삭제
  (circuitbreaker_open query 자체가 부재, §0-2 — "배선하면 되는" 문제가 아니라 현재
  파이프라인에 CB 메트릭이 전혀 없음).

## 9. calibration 게이트 (readiness=draft, live_allowed=false 유지 사유)

1. `target_rps=70`이 restaurant Hikari 10(최소풀)·Tomcat 스레드를 order 5xx 임계까지
   실제로 밀어내는지 첫 run에서 확정 필요 — F02-P의 companion은 50rps로 restaurant를
   흔들기만 했지 order 전량 502까지 겨냥한 값이 아니었으므로 그대로 재사용 불가, 반드시
   재보정.
2. `restaurant_error_rate`/`restaurant_p95` 절대 임계(현재 각각 0.1 gte, 별도 절대치 없음)는
   idle baseline 실측 없이 설정된 추정치 — calibration에서 baseline 대비 배율로 재확정
   권고.
3. §3의 러너 코드 조각(`APPROVED_APM_SERVICES`+`APPROVED_K8S_TARGETS`)이 main에 반영되고
   배포되기 전까지는 F24-Q도 F02-P도 restaurant 관측이 라이브에서 LiveProbeError로
   실패한다 — 이 반영이 F24-Q calibration의 **선행조건**(및 F02-P 자체의 숨은 결함 수리).
