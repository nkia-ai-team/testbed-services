# 문서 관리 규칙

## 목적

저장소 문서를 찾기 쉽고, 검토하기 쉽고, 유지보수하기 쉽게 관리합니다.
`docs/`의 문서는 단일 코드 주석이나 커밋 메시지를 넘어 프로젝트 동작,
결정 사항, 운영 절차, 참고 자료를 설명해야 합니다.

## 위치

- 오래 유지해야 하는 프로젝트 문서는 `docs/` 아래에 둡니다.
- `docs/README.md`는 문서의 진입점이자 목차로 유지합니다.
- 에이전트나 자동화 작업 지침은 일반 문서 관리 관례가 아닌 한 `docs/`가
  아니라 `AGENTS.md`에 둡니다.
- 임시 메모, 작업 초안, 자동 생성된 조사 결과는 검토를 거쳐 오래 유지할
  문서로 승격된 경우에만 `docs/`에 둡니다.

## 이름 규칙

- 파일 이름은 소문자 케밥 케이스를 사용합니다. 예: `api-contract.md`
- 문서를 훑어볼 때 도움이 되는 경우에만 설명용 접두어를 사용합니다.
  - `adr-`: 아키텍처 결정 기록
  - `runbook-`: 운영 절차
  - `ref-`: 안정적인 참고 자료
  - `spec-`: 제품 또는 구현 명세
- 다른 형식이 꼭 필요한 경우가 아니면 문서는 `.md`로 작성합니다.

## 구조

- 모든 Markdown 문서는 파일 최상단에 YAML front matter를 둡니다.
- front matter는 H1 제목보다 먼저 작성합니다.
- front matter에는 최소한 `title`, `status`, `owner`, `last_reviewed`,
  `tags`, `summary`를 포함합니다.
- `last_reviewed`는 `YYYY-MM-DD` 형식을 사용합니다.
- `tags`는 소문자 케밥 케이스를 사용합니다.
- front matter 예시는 다음과 같습니다.

```yaml
---
title: 문서 제목
status: Active
owner: project
last_reviewed: 2026-07-07
tags:
  - docs
summary: 문서가 설명하는 내용을 한 문장으로 요약합니다.
---
```

- 각 문서는 하나의 H1 제목으로 시작합니다.
- 문서 상단에 짧은 목적 설명을 둡니다.
- 지속적인 관리가 필요한 문서는 front matter의 담당자, 상태, 마지막 검토일을
  최신 상태로 유지합니다.
- 관련 문서는 상대 링크로 연결합니다.
- 비슷한 문서를 새로 만들기보다 기존 문서를 갱신하는 것을 우선합니다.

## 검토와 유지보수

- 동작, 운영 절차, 프로젝트 결정에 영향을 주는 문서 변경은 코드 변경처럼
  검토합니다.
- 문서가 설명하는 동작을 바꿀 때는 같은 변경 안에서 문서도 함께 갱신합니다.
- 외부 참조가 남아 있을 수 있는 문서는 삭제 전에 더 이상 유효하지 않음을
  명확히 표시합니다.
- 문서를 삭제하거나 대체할 때는 `docs/README.md`와 해당 문서를 가리키는
  링크도 같은 변경 안에서 갱신합니다.

## 상태 라벨

문서의 생명주기를 추적해야 할 때는 다음 라벨을 사용합니다.

- `Draft`: 아직 완성되지 않았거나 설계 중인 문서
- `Active`: 현재 기준이 되는 문서
- `Deprecated`: 기록을 위해 남겨두지만 더 이상 기준이 아닌 문서
- `Archived`: 추가 갱신을 기대하지 않는 과거 기록
