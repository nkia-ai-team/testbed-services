# 평가용 데이터 캡처·재생 설계

이 문서는 [평가·실험 설계](spec-evaluation.md)가 정의한 채점 방법론을 **재현
가능한 입력** 위에서 돌리기 위한 데이터 계층을 고정한다. 평가가 공정하려면
"같은 입력 → 대상 버전만 바꿔 점수 비교"가 성립해야 하는데, 테스트베드는
살아있는 서비스·부하·장애로 데이터가 매번 새로 생겨 그대로는 재현이 안 된다.
이 문서는 그 입력을 **한 번 녹화(캡처)해 두고 반복 재생(평가)** 하는 구조를
정의한다.

관련: [테스트베드 환경 설계](spec-testbed-design.md),
[시나리오 작성 규칙](spec-scenario-authoring.md),
[Lucida Next 데이터 수집 참조](ref-lucida-next-data-collection.md),
[Lucida Next AI 기능 참조](ref-lucida-next-ai-features.md).

## 1. 문제와 원칙

- **재현성.** 스트리밍 이상감지·클러스터·인시던트·RCA·챗봇을 평가하려면
  입력이 매 실행 동일해야 한다. 라이브 재실행은 타이밍 지터·부하 난수·GC로
  bit 단위 재현이 불가하므로 **점수용 평가에는 부적합**하다(통합·스모크용).
- **캡처는 데이터 계층에서.** 소비자가 여럿이고 각자 쿼리 패턴이 다르며 계속
  진화하므로 "응답 녹화/재생"은 깨진다. 대신 **저장소(DB)의 시간창 스냅샷**을
  얼리고, 평가 때 실제 쿼리를 날린다. 이러면 어떤 쿼리가 와도 일관되게 답한다.
- **오염 격리.** 파이프라인 각 단계는 공유 저장소에 되쓰기(write-back)를 한다
  (§3). 평가 중 단계를 재실행하면 원본이 오염되므로, 반드시 **격리된 일회용
  사본**에 복원해 돌리고 폐기한다.

## 2. 두 단계 — 캡처와 평가

평가는 두 단계로 분리한다. 이 둘을 섞으면 혼란해진다.

**Phase A · 캡처 (녹화)** — 시나리오당 한 번, 라이브 테스트베드.

1. 테스트베드에 에이전트·collector 설치(최초 1회).
2. **정상 구간** 확보 — 정상 부하로 장애 없이 유지. 이상감지가 "평상시"를
   학습하고 RCA가 비교할 기준을 만든다.
3. 장애 주입(시나리오 실행) → 인시던트 발생.
4. 회복까지 둔다.
5. `[정상 → 장애 → 회복]` 시간창을 3개 저장소에서 떠서 **케이스**로 저장하고,
   주입한 장애를 아는 상태에서 **골든 정답**을 기록한다.

**Phase B · 평가 (재생)** — 라이브 없이 무한 반복.

1. 케이스를 격리된 일회용 DB 세트에 복원한다.
2. 대상 기능 실행: 배치형은 조회, 스트리밍형은 재생(§5).
3. 결과를 골든 정답과 비교해 채점한다.
4. 격리 DB를 폐기하고 다음 케이스로 반복한다.

대상 버전(에이전트·모델)이 바뀌면 **Phase B만** 케이스 라이브러리에 다시
돌린다. 라이브 테스트베드를 재현할 필요 없이 전체 회귀 평가가 끝난다.

## 3. 파이프라인 저장 위치 (코드 기준)

`../lucida-next/backend` 확인 결과, 파이프라인 각 단계의 출력이 모두 durable
하게 저장된다. 캡처는 이 지점들을 시간창으로 뜬다.

| 경계 | 저장소 | 테이블/타깃 | 쓰는 곳 (file:line) |
|---|---|---|---|
| 원시 메트릭 | VictoriaMetrics | remote-write 시계열(`target_id`) | `ingest/writer/victoria_metrics.go:87` |
| 원시 트레이스 | ClickHouse | `lucida.otel_traces_local` | `ingest/writer/clickhouse.go:104` |
| 원시 로그 | ClickHouse | `lucida.lucida_logs_local` | `ingest/writer/clickhouse_lucida_logs.go:31` |
| 이상감지 이벤트 | ClickHouse | `lucida.lucida_events_local` | `store/events.go:56` (+검출기 7종) |
| 클러스터 | PostgreSQL | `event_clusters` (+CH `cluster_id` 태깅) | `store/clusters.go:82` |
| 인시던트 격상 | PostgreSQL | `incidents`/`incident_members`/`incident_timeline` | `store/incidents_judge.go:515` |
| 토폴로지 스냅샷 | PostgreSQL jsonb | `event_clusters.correlation_meta.topology` → `incidents.topology` | `topology_snapshot.go:22` |
| 토폴로지(라이브)·HostKin | 라이브 쿼리 | CH `host_connections` + PG `targets` | `signalgateway.go:131` |

RCA의 `IncidentSeed`(`rca/model/signals.go:19`)는 읽기 전용이며 PG `incidents`를
backing store로 쓴다(`signalgateway.go:71`).

## 4. 저장은 파일로, 평가 시 복원

케이스는 DB를 계속 띄워 두는 것이 아니라 **파일**로 저장한다. 모듈은 파일을
직접 읽지 못하고 DB에만 질의하므로, 평가할 때만 임시 DB로 복원했다 폐기한다
(파일 = 창고, 임시 DB = 재생 무대). 각 저장소의 native 덤프/복원 포맷을 쓰므로
**기존 모듈·API는 코드 수정 없이** 복원된 DB를 그대로 조회/재생한다.

케이스 디렉터리 구조(예):

```
case-01-inventory-lock/
  data/                     # 세계 스냅샷 (모든 소비자 공유)
    victoriametrics.export  # 메트릭
    clickhouse.parquet      # 트레이스·로그·이벤트·host_connections
    postgres.dump           # 클러스터·인시던트·targets·정책·토폴로지
  golden.rca.json           # RCA 정답(근본원인) — RCA 담당 소유
  golden.chatbot.json       # 챗봇 평가 질문+기대답변 — 챗봇 담당 소유
  golden.anomaly.json       # 이상감지 정답 이벤트 — 이상감지 담당 소유
  meta.json                 # 장애 시각·시간창·리드인 길이 등
```

**데이터는 공유, 골든만 따로.** 데이터(`data/`)는 한 벌을 모든 소비자가
공유한다. 소비자별로 다른 것은 데이터가 아니라 "문제+정답지"(골든)이며, 각
평가 담당자가 소유한다.

## 5. 두 종류의 데이터 — 슬라이스 vs 전체

"시간창 안의 것만 저장"은 대체로 맞지만 한 가지가 어긋난다.

| 종류 | 예 | 뜨는 방식 |
|---|---|---|
| 시간 데이터 | 메트릭·트레이스·로그·이벤트·알람·인시던트 | 시간창 `[t0,t1]`로 슬라이스 |
| 기준/인벤토리 데이터 | `targets`(자원 대장)·토폴로지·알람정책·서비스 카탈로그 | 시간창과 무관하게 그 시점 전체 스냅샷 |

인벤토리는 시간축이 없다. 챗봇의 "서버 몇 대냐"는 시간창 안의 *사건*이 아니라
현재 census다. 시간으로 자르면 인벤토리·현황·토폴로지 질문이 깨진다. 라이브로
계산되는 토폴로지·HostKin도 결국 CH `host_connections` + PG `targets`를 읽으므로
(§3), 이 참조 테이블은 반드시 전체를 뜬다.

## 6. 캡처 두 종류와 소비자

소비자는 여럿이고(RCA·이상감지·클러스터·인시던트·챗봇), 필요한 창의 성격이
다르다. 구분 축은 **"장애 유무"가 아니라 "창 길이·목적"** 이다. 장애는 양쪽 다
포함될 수 있다.

| 캡처 종류 | 길이 | 내용 | 주 소비자 |
|---|---|---|---|
| 짧은 집중 캡처 | 짧음(정상 리드인 + 장애 + 회복) | 장애 하나, 깨끗한 라벨 | RCA·이상감지·클러스터·인시던트, 챗봇의 장애 관련 질문 |
| 긴 운영 캡처 | 긺(수일~수주) | 정상 + 장애 섞임 | 챗봇(추세·인벤토리·현황 + 그 안의 장애 질문) |

- **리드인은 케이스별 가변**으로 `meta`에 명시한다. 짧은 집중 캡처는 이상감지
  정상 학습에 필요한 최소치(예: 1~2시간)만. 긴 과거는 긴 운영 캡처가 담당하며,
  모든 짧은 케이스에 수일치 리드인을 붙이는 낭비를 피한다.
- **재활용.** 긴 운영 타임라인을 한 번 캡처(장애 여러 개 주입)해 챗봇은 통째로
  평가하고, 그 안의 각 장애 구간을 잘라 짧은 컴포넌트 케이스로 파생할 수 있다.
- **트레이드오프.** 컴포넌트 평가(RCA 등)는 장애 하나·노이즈 적은 깨끗한
  조건에서 귀속이 명확하다. 긴 운영 캡처는 노이즈가 섞여 귀속이 어려워질 수
  있으므로, 보통 **깨끗한 짧은 케이스와 현실적인 긴 캡처를 둘 다** 둔다.

### 모듈별 입력·채점 컨셉

| 모듈 | 입력 | 출력 | 채점 컨셉 |
|---|---|---|---|
| 이상감지 | 원시 스트림(재생) | 이상 이벤트 | 정답 이상과 비교 — 제때 맞게 잡았나 + 오탐 안 냈나 |
| 클러스터 | 이벤트 집합(조회) | 이벤트 그룹 | 같은 사건 이벤트를 하나로 잘 묶었나 |
| 인시던트 격상 | 클러스터·이벤트(조회) | 인시던트(시각·범위·심각도) | 제때 격상 / 과·미격상 여부 |
| RCA | 인시던트+토폴로지+원시(조회) | 근본원인 | 원인 자원·도메인을 맞췄나 |
| 챗봇 | 위 전체 + 인벤토리·추세 | 자연어 답변 | 질문별 기대답변과 비교 |

**격리 vs 종단.** 각 단계에 앞 경계의 *골든* 입력을 주면 그 단계만 격리 채점해
귀속이 명확하다. 앞 단계의 *실제* 출력을 주면 종단(누적) 성능을 본다. 인프라는
격리를 지원하게 지으면 종단은 스위치 하나로 따라온다.

## 7. 챗봇 평가

챗봇(Polestar AI)은 자연어로 **무엇이든** 물어보므로(open-ended), 특정 테이블만
골라 뜨면 깨진다. 데이터는 §5 원칙대로 "세계 통째로"를 다른 소비자와 공유한다.
챗봇 고유로 따로 두는 것은 데이터가 아니라 질문+정답지(`golden.chatbot.json`)뿐이다.

- **입력.** 모든 소비자의 데이터 + 인벤토리·추세. 복원된 DB를 조회하는
  배치형이다(스트리밍 재생 불필요).
- **캡처.** 추세·인벤토리·현황 질문은 긴 맥락이 필요하므로 **긴 운영 캡처**
  (정상+장애 섞인 수일~수주, §6)를 쓴다. 장애·원인·알람 질문은 긴 캡처 안의
  장애 구간(또는 거기서 파생한 짧은 케이스)에서 답한다.
- **골든.** `golden.chatbot.json` = 질문 + 기대답변 세트. 챗봇 담당 소유.
  질문마다 어느 캡처를 겨냥하는지 함께 명시한다.
- **채점.** 질문별 기대답변과 비교한다. 자연어 답변이므로 구조화 필드는
  결정론, 표현이 다른 부분은 의미 등가로 채점한다([평가·실험 설계](spec-evaluation.md) 참조).

질문 유형과 캡처 매핑(예):

| 질문 유형 | 예 | 겨냥 캡처 |
|---|---|---|
| 인벤토리·현황 | "결제서비스 서버 몇 대?" | 긴 운영 캡처(기준 데이터 전체) |
| 추세 | "지난 7일 결제 지연 추세" | 긴 운영 캡처(긴 리드인) |
| 장애·원인 | "그 인시던트 원인 요약해줘" | 장애 구간(짧은 케이스 또는 긴 캡처 내) |
| 알람·임계치 | "지금 임계치 넘은 자원?" | 해당 시점 창 |

## 8. 스트리밍 재생

배치형(RCA·클러스터·인시던트·챗봇 대부분)은 복원된 DB를 그대로 조회한다.
스트리밍 이상감지는 DB를 폴링하지 않고 Kafka 토픽 `lucida.metrics`를 소비하므로
(`streamanomaly/runner/main.go:469`), 얼린 원시 값을 같은 토픽에 시간순으로
재발행하는 **재생기**가 필요하다. 감지기는 실서비스처럼 소비해 이벤트를 낸다.

**결정론(시계).** 감지의 핵심 판정(기준선·밴드·z-score·계절 버킷·에피소드
시작)은 데이터 타임스탬프(event-time) 기준이라 재생 시 그대로 재현된다
(`detector.go:900,971,1087`). 예외는 에피소드 **종료(close)** 로, 30초 벽시계
스윕이 KeepFiring 8분을 실제 시각으로 센다(`main.go:409` → `runtime.go:449`).
배속 재생 시 어긋나므로, 이미 있는 가상 시계 훅 `WithClock`(`detector.go:303`,
실행부 미연결 `:336`)을 데이터 시각으로 연결해야 종료까지 완전 재현된다.

## 9. 트레이너·시드 고정 (평가 모드)

스트리밍 감지 결과는 데이터만으로 정해지지 않는다. 별도 Python 트레이너
`ai-trainer`가 감지기 상태 스냅샷을 학습해 보정값 4종(계절·드리프트·임계치·분포)을
PG `trainer_stream_*` 테이블에 저장하고, 감지기가 주기적으로 reload한다. reload는
스트림 도중 임계치·기준선을 바꾼다. 평가 재현을 위해 다음을 고정한다.

- **reload OFF:** `STREAM_ANOMALY_TRAINER_RELOAD_SEC=0`,
  `STREAM_ANOMALY_MODEL_RELOAD_SEC=0`.
- **트레이너 정지:** `ai-trainer` 컨테이너 미기동(또는 `AUTOLOOP_AGENTS`에서
  `stream-anomaly` 제거).
- **모델·트레이너 테이블 active 행 고정**, `valid_until_ms` 만료 주의.
- **시드/백필 고정 또는 off:** `STREAM_ANOMALY_SEED_FROM_VM`,
  `_SEASONAL_BACKFILL`, `_SPIKE_SEED`.
- **coverage/detection-policy PG 행 고정** (`coverageRefreshLoop` 60초, env로
  못 끔 → 데이터로 고정).

**train/test 분리.** 모델은 해당 케이스의 장애 데이터로 학습되면 안 된다(누수).
그 케이스 이전/무관 데이터로 학습된 버전으로 고정한다. 골든 이상 라벨은 감지기
출력이 아니라 시나리오에서 나온 진짜 정답이어야 한다(순환논리 방지).

## 10. 콜드스타트 vs 웜스타트

감지기를 빈 상태로 시작할지, 학습된 상태를 얹고 시작할지의 선택.

| 방식 | 방법 | 재현성 | 적합 |
|---|---|---|---|
| 콜드스타트(기본) | 시드·reload·트레이너 전부 off, 빈 상태로 시작해 얼린 정상→장애만 재생. 스스로 평상시 학습 | 입력이 스트림 하나 → 완전 재현 | 급성 장애(스파이크·z-score·변화점) — 대부분의 시나리오 |
| 웜스타트 | 특정 모델·상태 스냅샷을 케이스에 함께 얼려 고정하고 시작 | 스트림+모델+스냅샷 모두 고정 필요 | 계절성(요일/시간대 장기 패턴) — 짧은 재생으로 못 만드는 경우 |

콜드스타트는 21일치 계절 패턴을 만들어내지 못한다. 계절성에 의존하는
시나리오만 웜스타트로 계절 프로필을 고정한다.
