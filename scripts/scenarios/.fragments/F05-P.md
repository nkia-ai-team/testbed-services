# F05-P — worker node memory pressure → 다서비스 eviction (live 승격 patch fragment)

정적 상세화 결과. 공유 파일은 직접 수정하지 않고 orchestrator가 이 fragment를
원자 적용한다. **라이브 주입·실측 없음** — eviction 강도는 adaptive memhog ladder로
calibration에서 확정한다.

## 요지
- 공통 인프라 root의 다서비스 merge 구조(F09-R/F15-P 계열). tb-w2
  (192.168.122.11)의 노드 메모리를 kubelet hard-eviction 임계
  (`memory.available<100Mi`)까지 밀어 **동거 pod 여럿을 evict** → 재스케줄 공백
  동안 commerce·food 여러 API 동시 실패.
- executor: `host.stress`에 **memhog 모드 신설**(profiles/host_stress_executor.py에
  **직접 구현 완료**). stress-ng 부재 → python이 $mib MiB를 anonymous 할당·전 페이지
  터치 후 hold. setsid 세션·pgidfile·`trap kill 0`·negative PGID cleanup(F09-R cpu
  모드와 동일 생명주기). **CPU 미점유**로 F09-R과 표면 분리.
- **실행 시점 배치 확인 게이트(fail-closed)**: commerce에 nodeSelector/affinity가 없어
  cohort가 재스케줄마다 이동(07-20 실측: cart pod suffix가 프로브 사이 변경됨). memhog
  preflight/run이 **노드 로컬 crictl로 현재 pod 목록을 재확인**하고 기대 cohort
  prefix(gateway·cart·inventory·redis·kafka·payment)가 실제 동거하지 않으면 **1바이트도
  할당 전 거부**. 노드 하드코딩 대신 런타임 검증.
- **stop-loss**: abort에 **pod_ready 미사용**(의도된 eviction이 pod_ready를 흔드는 건
  본질). 대신 node NotReady(노드 전면 붕괴)·stateful 컨테이너 restart 폭주(예산)로 중단.

## 실측 근거 (07-20 read-only)
- tb-w2 cohort(crictl pods, Ready): testbed-gateway, testbed-cart, testbed-inventory,
  testbed-kafka-0, testbed-redis(commerce) + testbed-payment(food) + testbed-ledger,
  testbed-nginx(banking). → 다도메인 동거 확인, F05-P 병합 root 성립.
- tb-w2 = **4 cpu, memory allocatable 12,113,304Ki(~11.55Gi)**, 현재 committed limits
  9088Mi(76%)/requests 3770Mi(31%). `free -m`: total 11929, used 3944, **available ~7985**.
- kubelet `evictionHard.memory.available=100Mi`, `evictionPressureTransitionPeriod=5m`.
- 워커 tooling: stress-ng **없음**, python3·setsid·yes·crictl(sudo) 있음.
- ladder MiB(5500/7000/8500)는 available 헤드룸(~7985) 기반 **정적 추정** — eviction
  무릎(7000~8500 사이)은 라이브 calibration에서 확정(잔여 조건).

### 지뢰 대응 근거
- **timeout(8m) > min_hold(4m)+settle(30s)**; ladder 3단 ≤ max_injection 30m.
- **abort에 pod_ready 없음**(지뢰 #4): 의도된 eviction·재스케줄이 pod_ready를 흔드는
  것은 시나리오 본질. 사용자 영향은 success의 checkout_5xx/entry, 안전은 node_ready로
  판정. 셀프 트리거(eviction의 pod NotReady 전파)로 조기 abort하지 않음.
- **stop-loss = node NotReady(전면 붕괴) + stateful(kafka) restart≥3(OOM 폭주 예산)**.
  깨끗한 kubelet eviction이 아니라 시스템 OOM으로 넘어가면 kafka 같은 stateful이
  재시작 → 예산 초과로 중단. 2틱(빠른 안전 중단).
- **escalate=단일 조건**(node_mem_util<92일 때만 MiB 상향). 노드 메모리 압박·eviction은
  발현 지연 → min_hold 4m 여유.
- **must_rule_out=압박 없이 증상만**(node_mem_util<50인데 5xx만) → 다른 원인, ruled_out.
- **cleanup**(지뢰 #9): memhog는 negative PGID kill+사망 확인(host_stress_executor.py
  memhog `cleanup`/`recovery` 검증 경로). env/limit 원복 없음(메모리만 점유, 파일·설정
  미변경) — 멱등.
