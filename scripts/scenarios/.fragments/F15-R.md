# F15-R — external PG mock 429↔200 반복 전환 flapping (장애→회복→재발)

정적 상세화 산출물. **라이브 주입·실측 미수행.** 기대 정답 = episode 경계 인지
(reopen이 아닌 **재발**). mock.expectation 실행기 재사용(F01-H 전례) + episode
worker(F15-R flap foundation 기존). episode 간격은 incident judge 시계를 감안해
**파라미터화**하고 보수적 기본값을 제안. 실측 확정 항목은 "첫 calibration에서 확정".

준비도 판정: **design-complete / live-blocked**. F15-T1과 동일한 timeline.compose
실행기 dispatch blocker + query 레지스트리 통합 + 첫 calibration 미실측. 아래 패치는
소유자 적용 체크리스트.

## 1. 잔여 조건 (live 승격 전 필수)

1. **timeline.compose 실행기 dispatch** — F15-T1 fragment §1.1과 **동일 공유 blocker**.
   dispatcher 실행기가 scenario_id로 F08-H(compose)·F15-R(flap)·F15-T1(dual)을 분기해야
   `timeline.compose` 한 profile이 셋을 서빙한다. flap 모듈은 이미 있으므로 dispatch에
   flap 경로만 연결하면 된다.
2. **query 레지스트리 통합.** F15-R 관측이 쓰는 3개 query id는 아직
   `observation_query_candidates.py`("awaiting runner registry integration")에만 있고
   `registry/queries.json`에 없다: `scenario.mock_flap_episode`,
   `scenario.mock_flap_fault_active`, `business.order_duplicate_count_since_t1`.
   live 전에 이들을 `queries.json`으로 승격해야 한다(§5). adapter(business_probe,
   database)는 이미 등록됨.
3. **첫 calibration 확정 항목**(실측 금지):
   - **episode 간격**: judge 시계(cluster close 3m·judge 보류 10m·LLM 유예 최대 30m,
     `spec-eval-data-capture.md §8.2`)를 감안해 두 episode 사이 정상 구간이 앞
     episode를 **완전히 close**시키고 judge가 **재발(신규 인시던트)** 로 인지하도록
     충분히 길어야 한다. 보수적 기본값: **장애 300s → 정상 720s → 장애 300s**
     (episode_count 2, fault_hold 300s, recovery_gap 720s). 판정 근거: cluster close
     3m + judge 보류 10m = 13m < 정상 720s(12m)… **720s는 13m 하한에 근접하므로
     경계.** 첫 calibration에서 (a) 720s로 episode 2가 별도 인시던트로 뜨는지,
     (b) 부족하면 LLM 유예(30m)까지 감안해 900–1200s로 상향하는지 확정한다.
     recovery_gap은 **강도가 아니라 정합성(경계 인지) 파라미터**이므로 ladder
     대상이 아니고 고정한다.
   - **fault_hold 강도**: 각 장애 episode가 (a) checkout 사용자 영향을 내고
     (b) 스트리밍 이상감지가 별개 episode로 잡기에 충분한 최소 길이. 첫 calibration이
     240s로 충분한지 300s가 필요한지 확정.
   - **companion 부하**: F01-H 동형 north-south 35rps로 각 429 episode가 checkout
     502를 유발하는지 확정(F01-H 선례상 성립 예상).

## 4. timeline_flap_executor.py bounds 확장 (공유 foundation — 정확 diff)

judge 시계 기반 episode 분리를 위해 bounds 상향. `validate()` 내:

```
- or not 30 <= params["fault_hold_seconds"] <= 120
- or not 60 <= params["recovery_gap_seconds"] <= 300
+ or not 30 <= params["fault_hold_seconds"] <= 360
+ or not 60 <= params["recovery_gap_seconds"] <= 1200
```

근거: 장애 episode 최대 6m(360s), 정상 간격 최대 20m(1200s)까지 허용 →
LLM 유예 30m 이내에서 판정 근거를 유지하며 재발을 별 인시던트로 분리할 여지 확보.
episode_count는 `==2` 고정 유지(변경 없음). 나머지 스크립트/worker 로직 불변.

## 7. expected_anomalies (탐지/클러스터/인시던트 — episode 경계)

- **탐지**: 각 장애 episode 동안 external payment 429 급증 + checkout 502/5xx 상승,
  episode 사이 정상 구간에는 신호 소멸(정상→이상→회복→정상 사이클 2회).
- **클러스터**: episode 1과 episode 2가 **각각 독립 클러스터로 open·close**.
  앞 episode는 정상 구간(recovery_gap) 동안 close 조건(3m 무활동) 충족으로 닫힘.
- **인시던트**: 두 episode는 **같은 외부 원인(PG 429)의 재발**. 기대 정답 =
  episode 2가 episode 1의 **reopen이 아니라 신규 인시던트(재발)** 로 생성. 두 episode
  사이 정상 구간과 동일 root cause가 결정적 구분 근거. t1=episode 1 fault 시작,
  t2=episode 2 fault 해제(정상 복귀). order_duplicate_count=0으로 F14-R(비멱등 중복)
  오판과 분리.
