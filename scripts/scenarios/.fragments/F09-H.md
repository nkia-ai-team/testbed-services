# F09-H — order JVM GC pressure (live 승격 patch fragment)

정적 상세화 결과. 공유 파일(catalog.json, registry/*.json, manifests/, tests/)은
**직접 수정하지 않고** 이 fragment의 패치를 orchestrator가 원자 적용한다.
라이브 주입·실측은 수행하지 않았다 — 강도는 adaptive Xmx ladder로 calibration에서 확정.

## 요지
- 주입 표면: `order` deployment의 `JAVA_TOOL_OPTIONS`에 `-Xmx/-Xms` 축소를 얹어
  재빌드 없이 heap을 좁힌다(F08-P의 SPRING_APPLICATION_JSON k8s.env 패턴 재사용).
  좁은 heap → 잦은 stop-the-world → order 전 endpoint p95 주기적 급등.
- executor는 기존 `k8s.env`(`k8s_env_executor.py`) 재사용. **F09-H는 이미
  `APPROVED_TARGETS`(rca-testbed-commerce/testbed-order/order-service)·
  `APPROVED_KEYS`({"JAVA_TOOL_OPTIONS"})에 등록되어 있어 executor 코드 변경 불필요.**
- controller = calibration + adaptive_ladder(heap 256→208→176 MiB 3단). companion =
  load.north_south 20rps.
- **OOMKill 금지 게이트**: restart_count==0 유지가 success 조건, restart_count>0은
  must_rule_out(=F05-R OOM loop와 구분 붕괴 → 케이스 무효). Xmx는 컨테이너 limit
  1Gi 내에서 pre-OOM pause 대역만 탐색.
- 관측: order p95/error(APM, commerce-order 기허용) + restart_count + jvm.*(메타).

## 실측 근거 (07-20 read-only)
- `testbed-order` = tb-w3, resources cpu 200m/500m, **memory 512Mi/1Gi**.
- 현재 `JAVA_TOOL_OPTIONS=-javaagent:/opt/apm/opentelemetry-javaagent.jar` (Xmx 미지정
  → 컨테이너 인지 기본 heap ~25% of 1Gi). baseline env = 아래 11개 exact snapshot
  (F08-P가 라이브 검증한 order env와 동일 배열).
- ladder 절대값(256/208/176 MiB)은 정적 추정 — **OOM 직전 pause 대역은 라이브
  calibration에서 확정**(잔여 조건).

### 지뢰 대응 근거
- **timeout(8m) > min_hold(4m)+settle(30s)**, ladder 3단 × 8m ≤ max_injection 30m.
- **escalate=단일 조건**(p95 미도달 시에만 heap 추가 축소). GC pause 발현 지연 →
  min_hold 4m 여유(F07-P/F08-P 교훈).
- **abort 8틱**: 각 ladder 승급이 env 재패치 → rollout blip(pod_ready 30~60s false).
  F08-P/F09-P 선례대로 8틱(2분) 미만이면 자기 rollout을 abort로 오인. pod_ready
  abort 허용됨(F09-H는 OOM 없으면 pod 유지 — F05-P와 달리 eviction 의도 없음).
- **must_rule_out=OOM 경계**: restart_count>0이면 즉시(1틱) ruled_out → STOPPING.
  OOMKill로 넘어가면 F05-R과 구분 붕괴. success의 `no-oom`(restart==0)과 이중 방어.

## 8. 잔여 조건 (라이브 승격 전)
1. **runner(109) 재빌드 불필요** — F09-H 관측은 전부 기존 query id(apm_service_p95/
   error_rate, container_restart_count, pod_ready, entry/target_health, achieved_rps)만
   사용. live_probes allowlist에 `commerce-order`·`testbed-order`는 F08-P에서 이미 추가됨.
   ✅ 신규 runner allowlist 없음.
2. **ai_coverages precondition**: jvm.* 4종(07-15 등록) + commerce-order APM p95/error가
   `ai_coverages`에 (target×metric) 등록되어 있어야 distinguishing_evidence의 jvm 신호가
   이상감지에 잡힌다. 승격 전 선행 확인(spec-scenario-load §4.1-1 coverage gate).
3. **pre-OOM pause 대역 calibration**: 256/208/176 MiB가 실제로 (a) p95 톱니를 만들고
   (b) restart_count 0을 유지하는지는 라이브 calibration 1회로 확정. OOM이 나면 ladder
   하한을 올려 재정의(관측에 golden 맞추지 않음 — 주입 강도 조정).
