# Fragment: F03-H — order servlet thread pool 고갈 (정적 상세화)

**준비도 판정: NOT READY — 승격 보류(partial 유지).** 라이브 주입 없이 정적 판정.
카탈로그 profiles=`["load.east_west"]`, load_mode=`adaptive`. **controller 미추가**(compile-plan은
controller 항목에 대해 readiness==ready를 강제하므로, controller를 넣는 순간 live로 승격됨 —
아직 성립하지 않는 시나리오를 live로 올리지 않기 위해 controllers.json/catalog는 건드리지 않는다).

## 왜 지금 승격 불가인가 (근거)

1. **집행기 정면 차단(정본).** `profiles/load_east_west_executor.py`가 F03-H를 명시적으로 거부한다:
   ```python
   if scenario_id == "F03-H":
       raise ExecutorError("F03-H blocked: no slow read-only order endpoint isolates servlet threads from DB")
   ```
   east-west 부하로 order servlet(tomcat worker) 스레드풀을 "깨끗하게" 포화시키려면, DB에 닿지
   않으면서 스레드를 오래 점유하는 order 엔드포인트가 필요하다. 그런 엔드포인트가 없으면
   부하는 결국 DB(order_schema)로 흘러 이 시나리오가 F03류 DB 경합/커넥션풀 시나리오로
   변질된다(집행기 저자의 차단 사유).

2. **엔드포인트 실측(order-service).** `OrderController`의 경로는
   `POST /`, `POST /checkout`(cart→pricing→inventory→payment 5-hop), `GET /`, `GET /stats/daily`,
   `GET /{id}`뿐이다. 인위적 sleep/blocking 엔드포인트는 없다. GET 계열은 전부 order DB를
   친다 — 스레드풀만 고립 포화시키는 read-only slow-call 표면이 부재.

3. **관측 공백(과제 명시).** tomcat executor 메트릭(active/queue/max threads)은 수집 파이프라인에
   없다. 그래서 스레드풀 고갈은 **직접 관측이 불가**하고 trace 대기·latency·JVM thread 계열로만
   간접 추정된다. success/abort 게이트를 걸 1차 원인 신호가 없다.

## expected_anomalies precondition (성립 시 걸어야 할 제약 — R9/§4.1)

승격을 시도하려면 golden `expected_anomalies`를 아래 제약 위에서만 걸 수 있다:

- **coverage gate**: stream anomaly는 `ai_coverages`에 (target×metric)으로 등록된 것만 처리한다.
  tomcat 스레드 메트릭은 미수집이므로 기대값을 tomcat executor 지표에 걸 수 없다. 대신
  등록 확인된 축에만 건다:
  - `signal: trace` — order→downstream span **대기/큐잉 시간** 증가(진입 요청이 worker 배정을
    기다리는 지연). precondition: 해당 trace 지표의 ai_coverage 등록.
  - `signal: metric` — `apm_service_p95(commerce-order)` 상승 + `apm_service_error_rate(commerce-order)`
    (스레드 고갈 시 진입 503/타임아웃). precondition: commerce-order APM coverage(이미 허용).
  - `signal: metric` — `jvm thread` 계열(가용 시). precondition: JVM thread coverage 등록 확인.
- **로그축은 ERROR만 즉발**: log anomaly `novel_template`은 severity ERROR(≥17)만 즉시 이벤트다.
  스레드풀 고갈 시 order가 **ERROR 레벨** 로그(예: `RejectedExecutionException`/커넥터 큐 초과)를
  실제로 찍는지 라이브 전 확인해야 로그축 기대값을 걸 수 있다. WARN 신규 템플릿은 즉발 아님 →
  기대값 금지.
- **precondition 필수 기입**: 위 각 항목의 관측 전제(coverage 등록·알람 정책)를 golden에 명시하지
  않으면 "왜 안 잡히는지" 추적 불가(§3).

## 승격 경로(둘 중 하나 — 이 태스크 범위 밖, parent 결정)

- **경로 A(앱 변경 필요, 현재 금지):** order-service에 DB 비접촉 slow read-only 엔드포인트를
  추가(재빌드·재배포). 그러면 load.east_west 집행기의 F03-H 차단을 해제하고 계단식 강도로
  스레드풀을 고립 포화 가능. **금지된 배포/재빌드를 수반하므로 지금은 불가.**
- **경로 B(표면 재정의):** east-west 슬로우콜 대신 **k8s.env로 `SERVER_TOMCAT_THREADS_MAX`를
  축소**(F03-P와 동형 config 표면, 재빌드 없음) + north-south checkout 부하로 소형 스레드풀
  포화. 카탈로그 profiles를 `["k8s.env","load.north_south"]`로 재정의해야 하고(과제가 명시한
  east-west Job 설계와 상충), 관측은 여전히 order APM p95/error_rate·trace 대기 간접 신호에
  의존. 채택 시 F03-P 프래그먼트와 동일 골격으로 상세화 가능.

## 조치 요약

- **파일 변경 없음**(catalog/controllers/profiles/manifests/executor 무수정). 집행기 차단·관측
  공백은 설계 결함이지 이 프래그먼트가 해결할 대상이 아님.
- 잔여 조건: (경로 A) order slow 엔드포인트 신설 + 배포, 또는 (경로 B) parent 승인 하 표면
  재정의 + tomcat 스레드 축소 실측. 어느 쪽이든 tomcat/JVM thread coverage 등록이 선행 precondition.
