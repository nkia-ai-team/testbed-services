# F08-G live 승격 fragment — distractor rollout(무관 변경) + banking Oracle lock 동시

## 준비도 판정
**정적 ready (live 등재 가능), 단 신규 profile `timeline.multi`의 라이브 검증 미실시.**
기존 `timeline.compose`는 F08-H 전용(rollout annotation + mock 429)이라 (notification 배포 + Oracle lock) 조합을 표현 불가 → 신규 executor 필요. 이 fragment가 그 executor + `timeline.multi` profile 등록을 소유한다.

근거:
- 신규 `profiles/timeline_multi_injection_executor.py` 작성·pytest 5건 통과·bash `-n` 통과.
- Oracle row `commerce-settlement` 실측 존재(F01-P fragment 참조). distractor `testbed-notification` deployment는 commerce 네임스페이스에 실존(kubectl get deploy 확인).
- golden 포인트: notification 변경은 checkout→payment→banking transfer 경로와 topology상 무관 → 변경이 원인 아님이 근거.

## 잔여 조건
1. **첫 live run = calibration-label**: (distractor rollout + Oracle lock) 조합의 cross-domain payment 증상 가시화 미검증.
2. **calibration ladder(강도=companion 부하)**: 락 이진이므로 강도는 north_south rps로 정의 — 첫 35rps 미가시 시 **35 → 70 → 110rps**(min_hold 4m).
3. **runner-side Oracle 태그 관측**(F01-P 잔여 #3과 동일 어댑터 요구, 태그 `rca-F08-G-oracle-lock`).
4. **manifest 재생성**(통합): profile_refs를 `timeline.compose`→`timeline.multi`로, controller/gate.live_allowed:true 반영.
5. **신규 profile 도입에 따른 테스트 카운트 파급**(아래 §counts).

## 지뢰 점검 결과
- (#4/#5 self-trigger, 최중요) **관측 pod_ready 대상 = `testbed-payment`(fault-path), 절대 `testbed-notification`(distractor) 아님.** distractor rollout은 notification pod를 의도적으로 churn시키므로, notification pod_ready를 abort/success에 넣으면 셀프 트리거. Oracle lock은 payment pod를 재생성하지 않으므로 payment pod_ready==false abort 2틱 안전. compile-plan이 강제하는 `entry_status==0`∧`pod_ready==false` abort는 payment 대상으로 충족.
- (#10 주입 위치) rollout=kubectl(commerce), Oracle lock=kubectl exec(banking pod). 둘 다 scenario-runner(109) 로컬 오케스트레이션. offset 15s로 distractor 선행 후 실장애.
- (#9 cleanup) reverse order 강제: oracle release(서버 세션 `alter system kill session`+v$session 소멸 폴링) → rollout restore(annotation 원복+rollout status). 멱등: state_dir 부재 시 cleanup no-op.
- (#7 정합) `timeline.multi` live_supported:true, companion north_south live_supported:true.
- (#11 precondition) expected_anomalies: commerce-payment error/latency 계열이 `ai_coverages`에 (target×metric) 등록돼 있어야 stream anomaly 감지 가능 — 승격 전 확인. notification 배포 change 이벤트는 change ingestion에 기록되나 **장애 경로 밖**임이 핵심(무관 증거).
