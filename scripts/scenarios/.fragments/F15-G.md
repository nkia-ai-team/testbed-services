# F15-G live 승격 fragment — commerce PG lock + banking Oracle lock, 같은 checkout trace의 multi-root

## 준비도 판정
**정적 ready (live 등재 가능), 신규 profile `timeline.multi` 라이브 검증 미실시.**
executor·profile 자산은 F08-G fragment가 소유하는 `timeline_multi_injection_executor.py`를 공유(F15-G 계약 포함). 이 fragment는 **row·주입 순서 확정** + F15-G 고유 배선을 소유한다.

근거:
- multi-root 설계: checkout trace가 inventory(PG) ∧ settlement(Oracle) 둘을 건드림. 두 행을 **동시(offset 0)** 점유 → 각 root가 독립적으로 checkout 실패를 유발 가능 → 단일 원인 확정 부당(golden `has_single_cause:false`).
- row 확정:
  - PG: commerce `inventory_schema.inventory` `product_id=1` (F01-R 검증 행 재사용, 태그 `rca-F15-G-inventory-lock`).
  - Oracle: `BANKING.accounts` `id='commerce-settlement'` (F01-P 실측 존재, 태그 `rca-F15-G-oracle-lock`).
- 주입 순서 확정: **동시(둘 다 offset_seconds:0)** — "같은 checkout trace 동시 발생"의 multi-root 성질 표현. 선후 인과 없음이 골든 포인트.

## 잔여 조건
1. **첫 live run = calibration-label**: 두 락이 각각 충분히 checkout를 실패시키는지(그리고 하나 해제해도 다른 하나로 실패 유지되는지) 미검증. 이 "각 root 독립 충분성"은 multi-root golden의 핵심이라 calibration에서 반드시 확인.
2. **calibration ladder(강도=companion 부하)**: 35 → 70 → 110rps(min_hold 4m). 두 락 동시 점유하에 부하를 올려 양 root 증상 동반 상승 확인.
3. **runner-side 태그 관측**: PG 태그(rca-F15-G-inventory-lock, 기존 PG 어댑터로 관측 가능) + Oracle 태그(rca-F15-G-oracle-lock, F01-P 잔여 #3의 Oracle 어댑터 지원 필요).
4. **manifest 재생성**(통합): profile_refs `timeline.compose`→`timeline.multi`, controller/gate.live_allowed:true.
5. **golden 레코드**: `has_single_cause:false`, `expected_status: insufficient`, pattern `multiple_concurrent_anomalies`. golden 파일 작성은 evaluation-capture 단계 몫(이 fragment는 시나리오 정적 상세화까지).

## 지뢰 점검 결과
- (#4/#5 self-trigger) 두 락 모두 pod 재생성 없음 → 관측 pod_ready(commerce payment fault-path) `==false` abort 2틱 안전. compile-plan 강제 gate(entry_status==0 ∧ pod_ready==false) 충족.
- (#3 escalate=ALL) success는 **두 태그 모두 존재** ∧ checkout 영향(AND) — 한쪽 root만으로 성급히 success 확정 방지.
- (#9 cleanup) reverse: oracle release(서버세션 kill+v$session 소멸) → pg release(pg_terminate_backend+pg_stat_activity 소멸). 멱등.
- (#10 위치) PG=ssh tb-runner NodePort 30432, Oracle=kubectl exec oracle pod. 둘 다 scenario-runner(109) 오케스트레이션.
- (#11 precondition) commerce inventory/payment 계열 메트릭 `ai_coverages` 등록 확인. multi-root라 두 계열 모두 stream anomaly coverage 필요.
