## 잔여 조건 (통합/실측 owner 몫 — 여기서 금지)
1. **첫 live run은 calibration-label로**: cross-domain 전파(banking Oracle lock → commerce payment 5xx)는 라이브 미검증. 첫 실행에서 `checkout_5xx`/payment error 가시화 확인 후 evaluation/golden 승격.
2. **calibration ladder(강도=companion checkout 부하)**: 락은 이진(강도 무단계)이므로 강도는 companion north_south rps로 정의한다. 1차 35rps에서 cross-domain 증상 미가시 시 **35 → 70 → 110rps** 순으로 상향(각 min_hold 4m). 이 ladder는 첫 run 결과에 따라만 적용.
3. **runner-side database 어댑터의 Oracle 지원**: 109 rca-scenario-runner의 `database.tagged_session_count` 관측이 Oracle `v$session.client_identifier`를 셀 수 있어야 함(F01-R는 PG application_name). runner live_probes/database adapter에 Oracle 경로 + `rca-F01-P-oracle-lock` 태그 허용 확인·재빌드 필요(레포 밖, 커밋 대상 아님).
4. **manifest 재생성**(통합): `generate-manifests.py`가 F01-P manifest의 controller/`prerequisite_gate.live_allowed:true`/actions live_enabled를 재생성.

## 지뢰 점검 결과
- (#4/#5 self-trigger) 관측 pod_ready 대상 = **commerce payment(fault-path)**. Oracle lock은 pod 재생성을 유발하지 않으므로 `pod_ready==false` abort 2틱은 셀프 트리거 아님(F01-R 동형). **compile-plan `validate_controllers`가 `entry_status==0` ∧ `pod_ready==false` abort를 강제**하므로 그대로 포함(정상).
- (#1 시간예산) timeout 10m > min_hold 4m + settle 30s + ramp. (#2 min_hold>ramp) 4m > 없음(fixed lock). (#3 lagging metric) APM error rate 지연 대비 min_hold 4m 여유(F07-P/F08-P 교훈).
- (#6 allowlist) north_south allowed_scenarios + tag_pattern + scenario_parameters에 F01-P 추가(아래). db.lock은 executor가 `CONTRACTS`로 단락 검증하므로 allowed_scenarios 추가 불필요하나 정합상 권장.
- (#7 controllers↔profiles) db.lock live_supported:true(기존), north_south live_supported:true(기존). 신규 profile 없음.
- (#8 freshness) tagged_db_sessions 30s, entry 15s, p95 120s(clock skew 10s 여유).
- (#9 cleanup) 멱등: `stop()` guarded, recovery `! alive`+row. **잔여 권고(비차단)**: Oracle cleanup은 client pid kill + PMON 롤백 의존 — 서버 세션 명시 kill/v$session 소멸 확인은 없음. F15-G용 신규 `timeline_multi_injection_executor.py`의 oracle release(`alter system kill session` + v$session 폴링)를 db_lock 경로에도 후속 반영 권장(단, db_lock 수정은 F01-R live hash 변경 위험 → 별도 창구).
